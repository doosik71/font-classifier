# Font Dataset Browser — 사용자 설명서 및 상세설계서

`scripts/font-dataset-browser.py`는 `data/annotation`에 저장된 annotation
정보를 이용해 `data/scan`의 원본 스캔 영상에서 완성형 한글 2,350자 낱글자
영상을 폰트 단위로 추출·정규화해 화면에 보여주는 Tkinter GUI 도구이다.

`scan-font-browser.py`가 **영상(페이지) 한 장** 단위로 annotation을
입력/저장하는 도구라면, 이 도구는 그렇게 저장된 annotation을 모아
**폰트(2,350자) 단위**로 실제 낱글자 영상이 올바르게 잘려 나오는지
검증하기 위한 것이다. `data/annotation`과 `data/scan`을 읽기만 하며
아무것도 쓰지 않는다 — annotation을 이용해 실제로 학습용 데이터셋을
파일로 잘라내는 단계는 아직 이 도구의 범위가 아니다(README "진행 상황"
참고).

이 문서는 두 부분으로 구성된다.

- **1부. 사용자 설명서** — 도구를 실행하고 화면을 읽는 방법
- **2부. 상세설계서** — 글자 추출/정규화 알고리즘, 폰트 그룹핑 규칙 등
  유지보수에 필요한 설계 정보

## 1부. 사용자 설명서

### 1.1 실행 방법

```bash
bin\font-dataset-browser.bat
```

또는 직접:

```bash
uv run python scripts/font-dataset-browser.py
```

`bin\font-dataset-browser.bat`는 다른 가상환경이 활성화되어 있어도 이를
무시하고 항상 프로젝트의 `uv` `.venv`만 사용하도록 `VIRTUAL_ENV`를 비운
뒤 실행한다 (`scan-font-browser.bat`와 동일한 방식).

### 1.2 화면 구성

```text
┌───────────────────────┬──────────────────────────────────────────────┐
│ 폰트 목록 (N)         │ 가  각  간  갇  갈  ...  (20열, 스크롤 가능) │
│ (초록=annotation 완비 │ [64x64 정규화 글자 영상 격자]                │
│  주황=일부 누락)      │                                              │
│                       │                                              │
│ 범례 설명             │                                              │
└───────────────────────┴──────────────────────────────────────────────┘
                    상태 표시줄 (하단)
```

- **왼쪽 – 폰트 목록**: `data/annotation`에서 읽은 폰트 이름을 가나다순
  으로 나열한다. annotation이 상당량(100자 이상) 누락되어 검증할 수 없는
  폰트는 목록에서 제외되며, 대신 콘솔에 오류로 출력된다(1.4절 참고).
  - **초록색**: 이 폰트는 2,350자 전체에 annotation이 있다.
  - **주황색**: 일부(100자 미만) 글자의 annotation이 아직 없지만, 나머지
    정보만으로도 확인이 가능하므로 목록에 포함했다.
- **오른쪽 – 글자 격자**: 폰트를 선택하면 원본 스캔 페이지와 동일한
  20열 격자로 2,350자를 순서대로 이어붙여 보여준다(페이지 경계가 25행
  마다 자연스럽게 일치한다). 각 칸은 64x64로 정규화된 실제 낱글자
  영상(그레이스케일)을 먼저 보여주고, 그 아래에 이 자리에 있어야 할
  글자를 작게 표시한다 — 영상을 먼저 보고 그 아래 정답 글자를 확인하는
  순서가 눈에 더 익숙하기 때문이다. 화면이 길어 마우스 휠이나 세로
  스크롤바로 스크롤한다.
- **글자 칸 테두리 색**: 정상적으로 추출된 칸은 영상 주위에 옅은 회색
  테두리만 표시되어 배경과 구분된다. 추출하지 못한 칸은 빈 칸으로 두되
  이유에 따라 빨간 테두리의 진하기를 다르게 표시한다.
  - **연한 빨강**: 이 글자의 annotation 자체가 없다(아직 작업되지 않은
    페이지, 또는 100자 미만의 작은 누락).
  - **진한 빨강**: annotation은 있지만 실제 칸에 잉크가 없다 — 이
    폰트에 해당 글자가 없어 빈 칸으로 인쇄된 것으로 보인다. 눈에 잘
    띄도록 진한 빨강으로 표시한다.
- **상태 표시줄**: 선택한 폰트 이름과 함께 "annotation 없음 N자, 빈
  칸(글자 없음) M자"를 보여준다.

### 1.3 사용 순서

1. 왼쪽 목록에서 확인하고 싶은 폰트를 클릭한다.
2. 오른쪽 화면에 2,350자가 격자로 표시될 때까지 잠깐(폰트 하나당 최대
   5장의 원본 영상을 열어 2,350칸을 처리하므로 1초 안팎) 기다린다.
3. 격자 위 각 칸의 실제 정규화 영상과 그 아래 예상 글자 레이블을 눈으로
   대조해 annotation(격자 좌표, 회전 보정, 첫 글자 등)이 올바른지
   확인한다. 어긋나 보이면 해당 페이지를 `scan-font-browser.py`에서
   다시 확인/저장한다.
4. 빨간 테두리 칸이 많다면 콘솔 출력을 함께 확인해 원인(annotation
   누락인지, 실제로 그 폰트에 글자가 없는 것인지)을 구분한다.

### 1.4 콘솔 메시지

이 도구는 화면에 다 담기 어려운 세부 정보를 콘솔(터미널)에 출력한다.

- 실행 시작 시, 폰트별로 annotation이 일부 누락되었지만 목록에 포함한
  경우 `[WARNING]`으로, 100자 이상 누락되어 목록에서 제외한 경우
  `[ERROR]`로 한 줄씩 출력한다.
- 폰트를 선택해 화면을 그릴 때, annotation이 없어 표시하지 못한 글자
  수와, 실제로 그 폰트에 글자가 없어 빈 칸으로 확인된 글자를 하나씩
  `[WARNING]`으로 출력한다(어떤 글자인지 알 수 있도록 문자와 인덱스를
  함께 보여준다). 콘솔 메시지는 영어로 출력한다.

이 메시지들은 annotation 데이터의 무결성을 점검하는 근거 자료이므로,
`bin\font-dataset-browser.bat`를 콘솔 창에서 실행해 두면 함께 확인할 수
있다.

## 2부. 상세설계서

### 2.1 폰트 그룹핑 규칙 (`font_dataset.build_font_entries`)

`data/annotation/*.json`을 모두 읽어 `font_name` 값이 같은 것끼리 묶는다.
README/scan-font-browser.md에서 확인했듯, 폰트 하나는 zip 안에서 연속된
최대 5장(500자 x 4장 + 마지막 1장)으로 구성되고 zip 경계가 항상 폰트
경계와 일치하므로, 같은 폰트의 페이지는 항상 같은 zip 안에 있다.

같은 폰트의 페이지들은 image_name(8자리 일련번호, 데이터셋 전체에서
겹치지 않고 스캔 순서를 그대로 반영) 오름차순으로 정렬한 뒤, 앞에서부터
차례로 `first_char_index`~`last_char_index` 범위를 글자 인덱스 → 페이지
매핑(`char_pages`)에 채운다. `dict.setdefault`를 쓰므로 이미 채워진
인덱스는 덮어쓰지 않는다 — 즉 **한 페이지가 중복 스캔되어 같은 글자
인덱스가 여러 페이지에 걸쳐 있으면, image_name이 더 작은(먼저 스캔된)
페이지가 그 글자를 차지한다.**

### 2.2 페이지 전체 누락 판단 (`MISSING_PAGE_THRESHOLD`)

폰트 하나(2,350자) 중 `char_pages`에 배정되지 못한 글자 수를
`missing_count`라 한다.

- `missing_count >= 100`(기본값)이면 페이지 한 장(최대 500자) 이상이
  통째로 비어 있다고 보고, 이 폰트를 목록에서 완전히 제외한다. annotation
  이 하나도 없는 페이지와, 어쩌다 몇 글자만 빠진 경우를 구분하기 위한
  기준이며, 500자짜리 페이지 하나가 통째로 빠지는 상황을 확실히 잡아내면
  서도 낱개로 흩어진 소수의 누락은 걸러내지 않도록 100이라는 값을
  선택했다.
- `0 < missing_count < 100`이면 목록에는 포함하되(annotation이 계속
  채워지는 중이므로) 주황색으로 표시해 불완전함을 알린다.
- 이 판단은 **annotation의 존재 여부만** 본다. annotation이 있는
  페이지에서 특정 칸에 실제 잉크가 없는 것(2.4절의 "빈 칸")은 여기 포함
  되지 않는다 — 그것은 페이지 누락이 아니라 폰트 자체에 그 글자가 없는
  것이기 때문이다.

### 2.3 글자 영상 추출/정규화 (`char_extract.extract_char_cell`)

annotation에 저장된 `grid`(칸 좌표)와 `rotation_deg`(회전 보정)를 그대로
사용한다 — `scan-font-browser.py`가 저장할 때 이미 사람이 확인한 값이므로
다시 추정하지 않는다. 페이지 영상은 저장된 각도로 한 번 회전시킨 뒤
(`Image.rotate(angle, resample=BICUBIC, expand=False, fillcolor=흰색)`,
scan-font-browser.py와 동일한 방식) 재사용하고, 그 위에서 칸 좌표로 잘라
낸다.

칸 하나에서 글자 하나를 만드는 절차:

1. `origin_x + col*cell_w` ~ `origin_x + (col+1)*cell_w` (Y축도 동일)
   범위로 칸을 잘라 흑백(그레이스케일)으로 변환한다.
2. Otsu 방법(`otsu_threshold`, 클래스 간 분산이 최대가 되는 임계값 탐색)
   으로 그 칸에 맞는 이진화 임계값을 계산한다. 스캔 농도가 영상마다 조금
   씩 다를 수 있어 고정 임계값 대신 칸별로 적응적으로 계산한다. 이
   임계값은 **바운딩 박스를 찾는 용도로만** 쓴다 — 화면에 표시하는
   최종 영상은 이진화하지 않는다(2.5절 참고).
3. 그 임계값보다 어두운 픽셀(잉크)의 바운딩 박스를 구한다. 잉크 픽셀이
   `MIN_INK_PIXELS`(5) 이하이면 이 칸에는 글자가 없다고 보고 `None`을
   반환한다(2.4절).
4. 바운딩 박스만큼만 그레이스케일 그대로 잘라내, 긴 변이 정확히 64px가
   되도록 **종횡비를 유지한 채** `Image.resize`로 확대 또는 축소한다.
   가로세로에 같은 배율을 적용하므로 종횡비와, 글자 크기 대비 획
   두께의 상대적 비율은 그대로 유지된다. 원본보다 작은 글자도 이
   배율만큼 확대되므로, 폰트마다 실제 인쇄 크기가 달라도(예:
   YDEchoM처럼 칸에 비해 작게 인쇄된 폰트) 화면에서는 비슷한 크기로
   비교할 수 있다 — 대신 절대적인 인쇄 크기 정보는 이 단계에서
   사라진다.
5. 64x64 흰색(255) 캔버스 중앙에 붙여 넣는다.

종횡비를 유지해야 하는 이유는 한글 폰트의 형태를 분류할 때 글자의
종횡비와 획 두께가 중요한 특징이기 때문이다(정사각형으로 강제로 늘이면
이 정보가 왜곡된다). 다만 이는 이 검증 도구의 결과물에 대한 이야기이며,
실제 폰트 인식 단계에서는 사용자가 사진에서 추출한 글자의 종횡비가
원본과 다를 수 있으므로 종횡비에 무관하게 인식하는 방법이 별도로
필요하다(README 참고, 향후 과제).

### 2.4 "빈 칸"의 두 가지 원인과 화면 표시

이 도구가 어떤 글자를 표시하지 못하는 데는 서로 다른 두 원인이 있고,
화면과 콘솔에서 구분해서 알려준다.

| 원인             | 조건                                                               | 화면 표시        | 콘솔 메시지                              |
| ---------------- | ------------------------------------------------------------------ | ---------------- | ---------------------------------------- |
| annotation 없음  | 해당 글자 인덱스가 `char_pages`에 없음(2.2절의 작은 누락)          | 연한 빨강 테두리 | 폰트 선택 시 누락 글자 수를 한 번에 출력 |
| 글자 없음(빈 칸) | annotation은 있지만 `extract_char_cell`이 `None` 반환(2.3절 3단계) | 진한 빨강 테두리 | 글자·인덱스별로 하나씩 출력              |

전자는 **아직 사람이 annotation을 만들지 않은 것**이고, 후자는 **그
폰트에 물리적으로 그 글자(예: 흔치 않은 종성 조합)가 없어서 인쇄 자체가
비어 있는 것**이다. 실제로 여러 폰트에서 `쏀`(인덱스 1298)처럼 특정
글자가 반복적으로 빈 칸으로 나타나는 것을 확인했다 — 폰트 파일에 그
글자의 자소 조합이 없는 경우로 보인다.

### 2.5 화면 배치와 렌더링 (`FontDatasetBrowser._render_font`)

2,350자를 원본 페이지와 같은 20열 격자로 이어붙여 스크롤 가능한 단일
캔버스에 그린다(500자 = 25행이므로 페이지 경계가 정확히 25행 배수와
일치해 별도 구분선 없이도 페이지 단위를 알아볼 수 있다). 칸 하나는 위에
64x64 글자 영상, 그 아래 9pt 글자 라벨(`LABEL_HEIGHT`=14px)로 구성된다
(`CELL_PITCH_X`/`CELL_PITCH_Y`) — 영상을 먼저 보고 정답 글자를 그 아래에서
확인하는 순서가 더 익숙하기 때문에, 영상을 라벨보다 위에 그린다.

폰트를 선택할 때마다 그 폰트가 걸쳐 있는 페이지(최대 5장)의 원본 영상을
zip에서 읽어 회전 보정까지 적용한 뒤 `page_image_cache`에 캐시해 두고,
2,350칸을 순회하며 같은 페이지 영상을 반복 사용한다. `ImageTk.PhotoImage`
객체는 가비지 컬렉션되지 않도록 `self.tk_images` 리스트에 보관한다(canvas
에만 그리고 참조를 버리면 Tkinter가 이미지를 화면에서 지워 버린다).

### 2.6 모듈 구조 요약

| 구성 요소                                                               | 파일                                      | 역할                                                                      |
| ----------------------------------------------------------------------- | ----------------------------------------- | ------------------------------------------------------------------------- |
| `otsu_threshold` / `extract_char_cell` / `CHAR_SIZE` / `MIN_INK_PIXELS` | `font_classifier/char_extract.py`         | 칸 하나에서 글자 하나를 64x64로 추출/정규화 (2.3, 2.4절)                  |
| `build_hangul_table` / `HANGUL_TABLE`                                   | `font_classifier/font_dataset.py`         | 완성형 2,350자 순서 (scan-font-browser.py와 동일 로직, 중복 정의)         |
| `FontEntry` / `build_font_entries` / `MISSING_PAGE_THRESHOLD`           | `font_classifier/font_dataset.py`         | annotation을 폰트 단위로 그룹핑, 글자→페이지 매핑, 누락 판정 (2.1, 2.2절) |
| `FontDatasetBrowser(tk.Tk)`                                             | `scripts/font-dataset-browser.py`         | 전체 GUI. 폰트 목록/색상 표시, 선택한 폰트의 2,350자 격자 렌더링 (2.5절)  |

`char_extract.py`/`font_dataset.py`는 `scripts/`가 아니라 `font_classifier/`
패키지에 있다 — `scripts/`에는 사용자가 직접 실행하는 진입점 스크립트만
두고, 여러 스크립트가 함께 쓰는 재사용 가능한 로직은 `font_classifier/`
패키지에 모아 관리한다(`grid_autocorrect.py`도 마찬가지이며, 이 도구뿐
아니라 `scan-font-browser.py`/`auto-correct-annotation.py`도 함께
사용한다). `GridParams`는 `font_classifier/grid_autocorrect.py`에서
그대로 가져와 사용한다(격자 좌표 계산 자체는 이미 annotation에 저장되어
있으므로 이 도구는 자동 보정을 다시 수행하지 않는다).

### 2.7 알려진 제한사항

- 폰트 그룹핑은 `font_name` 문자열이 정확히 같은지만 본다. 오탈자 등으로
  같은 물리적 폰트가 다른 이름으로 저장되어 있다면 다른 폰트로 취급된다.
- 이 도구는 annotation의 `grid`/`rotation_deg`를 검증 없이 그대로
  신뢰한다. 격자 자체가 잘못 저장된 페이지는 `scan-font-browser.py`에서
  먼저 바로잡아야 한다.
- 폰트 하나를 선택할 때마다 최대 5장의 영상을 열고 2,350칸을 매번 다시
  추출한다(디스크에 캐시하지 않음). 검증 도구로서는 충분히 빠르지만
  (폰트당 1초 안팎), 실제 학습 데이터셋을 만드는 단계에서는 결과를
  파일로 저장하는 별도 도구가 필요하다(README "진행 상황"의 다음
  과제).
