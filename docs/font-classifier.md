# Font Classifier (인식 앱)

학습된 폰트 인식 모델(`data/checkpoints/latest.pt`)로 **낱글자 하나**의
한글과 폰트를 인식해 결과를 보여 주는 Tkinter GUI 앱이다. README의 최종
목표("사용자가 제시한 영상에서 한글 글자를 추출하고 폰트를 인식")의 낱글자
단위 버전으로, `scripts/font-classifier.py`에 있다.

```bash
bin/font-classifier.sh        # 또는  uv run python scripts/font-classifier.py
```

## 1. 입력 고르기

왼쪽 패널의 두 탭 중 하나로 인식할 글자를 고른다.

### 데이터셋 탭

- 왼쪽 목록에서 폰트를 고르면 `data/dataset/<id>.png`의 2,350자를
  `dataset-browser.py`와 같은 20열 격자로 표시한다.
- 글자 칸을 클릭해 인식할 글자를 하나 선택한다(파란 테두리로 표시). 이미
  64x64로 정규화된 글리프이고 정답(폰트/글자)을 알고 있으므로, 인식 결과와
  함께 정답을 표시하고 맞았는지 색으로 구분한다.
- 그 폰트에 해당 글자가 없는 빈 칸(빨간 테두리)은 선택해도 인식하지 않고
  안내만 한다.

### 파일 탭

- "영상 열기…"로 임의의 이미지를 연다(화면에 맞게 축소해 표시).
- 글자 하나를 감싸는 사각형을 드래그해 고른다. 다중 글자 사진에서 한 글자만
  선택하는 용도다. 박스를 그리지 않으면(또는 "박스 지우기") 영상 전체가
  낱글자 하나라고 보고 그대로 쓴다.
- 고른 영역은 학습 때와 **똑같은** `char_extract.normalize_glyph`
  (Otsu 최소 바운딩 박스 → 긴 변 64px 정규화 → 흰 배경 중앙 배치)로 글자를
  뽑는다. 잉크가 거의 없으면 안내만 하고 인식하지 않는다.

### augmentation 체크박스

하단의 "augmentation 적용"을 켜면 `dataset_loader`의 학습용 augmentation
(약한 세트: 기하/광도/흐림/노이즈/JPEG/erasing)을 입력에 **한 번** 적용한
뒤 인식한다. 학습 입력과 똑같은 변형 경로라, 사진 노이즈에 대한 강건성을
눈으로 확인할 때 쓴다. 꺼 두면 [0,1] 정규화만 하고 그대로 넣는다. 어느
경우든 "모델에 실제로 들어간 영상"을 결과 패널 왼쪽 위에 미리 보여 준다.

## 2. 인식과 결과 표시

"인식 실행"을 누르면 `model.encode()`를 한 번 호출해 초/중/종성 logits과
폰트 logits을 얻고, 오른쪽 결과 패널에 다음을 표시한다.

- **한글 인식**: 제한 디코딩(2,350자 표, `decode_restricted`)과 개방
  디코딩(11,172자, `decode_open`) 결과를 모두 보여 준다. 정답을 아는
  데이터셋 입력이면 맞음(초록 ✓)/틀림(빨강 ✗, 정답 병기)을 색으로 구분한다.
- **폰트 상위 10위**: 폰트 logits에 softmax를 씌운 확률 상위 10위의 폰트
  이름과 확률을, "인식된 글자(제한 디코딩 결과)를 그 폰트로 쓴 64x64
  글리프"와 함께 나란히 표시한다. 예: 명조체 "가"로 인식하면 각 후보 폰트의
  "가" 글리프를 데이터셋에서 읽어 보여 준다. 데이터셋 입력이면 정답 폰트
  행을 초록 배경으로 강조하고, 정답이 10위 밖이면 실제 순위를 따로 알려 준다.

폰트 logit 인덱스 `i`는 `data/dataset/index.json`의 `id = i + 1`에
대응한다(학습 시 `dataset_loader`가 `font_label = id - 1`로 라벨을 만든 것과
같은 규칙). 이 매핑으로 logit → 폰트 이름/글리프 파일을 찾는다.

## 3. 설계 메모

- **정규화 일관성**: 앱과 학습이 반드시 같은 글자 정규화를 써야 입력 도메인이
  같아진다. 이를 위해 `char_extract.py`의 정규화 부분을 `normalize_glyph`로
  분리해 격자 칸 추출(`extract_char_cell`)과 앱의 임의 영역 추출이 그대로
  공유한다.
- **폰트 글리프 로딩**: 후보 폰트의 세로 PNG를 폰트별로 캐시해(`id → PIL`)
  반복 인식 시 다시 디코딩하지 않는다.
- **모델 로딩**: 폰트 클래스 수는 체크포인트의 폰트 분류기 가중치 크기에서
  직접 읽어, 데이터셋을 다시 스캔하지 않고도 모델을 구성한다.

## 4. 전제: 체크포인트와 데이터셋의 정합성

이 앱은 `latest.pt`의 폰트 클래스 순서가 현재 `data/dataset/index.json`의
`id` 순서와 **일치한다**고 가정한다(둘 다 `build_font_entries()`의
폰트 이름 정렬 순서에서 나온다). 두 산출물이 서로 다른 시점의 것이어서
폰트 집합/순서가 어긋나면, 자소(한글) 인식은 폰트 순서와 무관하므로 여전히
정확하지만 **폰트 이름 매핑이 틀어진다**(모델은 폰트 정체성을 학습했더라도
그 클래스 번호가 현재 `id - 1`과 달라진다). 폰트 인식 결과가 한글 인식에
비해 유독 나쁘면 가장 먼저 이 정합성을 의심하고, 현재 `data/dataset`으로
학습한 체크포인트인지 확인한다.
