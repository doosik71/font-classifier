# Scan Font Browser — 사용자 설명서 및 상세설계서

`scripts/scan-font-browser.py`는 `data/scan`에 담긴 폰트 스캔 영상을 눈으로
확인하고, 딥러닝 학습에 필요한 annotation(폰트 이름/글자 범위/격자 좌표/회전
보정각)을 사람이 직접 입력해 `data/annotation`에 저장하는 Tkinter GUI
도구이다.

이 문서는 두 부분으로 구성된다.

- **1부. 사용자 설명서** — 도구를 실행하고 annotation을 작성하는 방법
- **2부. 상세설계서** — 내부 동작 원리, 좌표 변환, 파일 형식 등 유지보수에
  필요한 설계 정보

## 1부. 사용자 설명서

### 1.1 실행 방법

```bash
bin\scan-font-browser.bat
```

또는 직접:

```bash
uv run python scripts/scan-font-browser.py
```

`bin\scan-font-browser.bat`는 miniforge/conda 등 다른 가상환경이 활성화되어
있어도 이를 무시하고 항상 프로젝트의 `uv` `.venv`만 사용하도록
`VIRTUAL_ENV`를 비운 뒤 실행한다.

### 1.2 화면 구성

```text
┌───────────────────────┬──────────────────────────────────────────────┐
│ ZIP 파일 (콤보박스)   │ 열/행/시작X/시작Y/칸너비/칸높이 (격자 조정)  │
│                       │ [격자 표시] [자동 보정] [기본값]             │
│ 이미지 목록           │           [폰트 이름 overlay, 상단 중앙]     │
│ (초록=저장됨          │              메인 캔버스                     │
│  빨강=미저장)         │        (영상 + 격자 + 예상 글자 overlay)     │
│                       │                                              │
├───────────────────────┤                                              │
│ Annotation            │                                              │
│  폰트 이름            │                                              │
│  첫 글자              │                                              │
│  마지막 글자: (자동)  │                                              │
│  회전 보정 각도       │                                              │
│  [Annotation 저장]    │                                              │
│  [다음 영상으로 이동] │                                              │
└───────────────────────┴──────────────────────────────────────────────┘
                    상태 표시줄 (하단)
```

- **왼쪽 위 – ZIP 파일**: `data/scan/*.zip` 목록. 하나를 고르면 그 안의 jpg
  파일 이름(8자리 숫자)이 아래 목록에 채워진다.
- **왼쪽 – 이미지 목록**: annotation이 저장된 영상은 **초록색**, 아직
  저장되지 않은 영상은 **빨간색**으로 표시된다.
- **오른쪽 위 – 격자 조정**: 격자의 열/행 개수와 시작 좌표(X, Y), 칸
  크기(너비/높이)를 직접 조정한다. `자동 보정` 체크박스가 켜져 있으면
  annotation이 저장되지 않은 영상을 선택할 때마다 시작 좌표와 회전 보정
  각도(Annotation 패널의 값)를 스캔 영상 분석으로 자동 추정한다 (1.7절
  참고). `기본값` 버튼을 누르면 자동 보정 여부와 관계없이 격자를 항상
  표준 기본값으로 되돌린다.
- **메인 캔버스**: 선택한 영상, 격자 overlay, (첫 글자를 입력했다면) 각 칸에
  들어가야 할 예상 글자를 파란 글씨로 함께 보여준다. 입력한 **폰트
  이름**은 영상 상단 중앙에 흰 배경 박스로 오버레이되어, 스캔 원본에
  인쇄된 폰트 이름과 나란히 놓고 바로 대조할 수 있다. 캔버스를 클릭하면
  상태 표시줄에 클릭한 위치가 몇 행 몇 열인지 표시된다(격자 검증용).
- **Annotation 패널**: 폰트 이름, 첫 글자, (자동 계산되는) 마지막 글자,
  회전 보정 각도를 입력하고 저장한다. 회전 보정 각도도 `자동 보정`이
  켜져 있으면 스캔 영상 분석으로 기본값이 채워진다.

### 1.3 격자(grid) 색상의 의미

메인 캔버스의 격자 선 색은 **현재 보고 있는 영상에 저장된 annotation이
있는지**를 나타낸다.

| 색상   | 의미                                        |
| ------ | ------------------------------------------- |
| 빨간색 | 이 영상은 아직 annotation이 저장되지 않았다 |
| 초록색 | 이 영상은 annotation이 이미 저장되어 있다   |

왼쪽 이미지 목록의 글자 색도 동일한 의미(빨강=미저장, 초록=저장됨)이며,
격자 색은 지금 보고 있는 화면에서 더 크고 분명하게 같은 정보를 보여주기
위한 것이다.

### 1.4 annotation 작성 순서 (기본 작업 흐름)

1. ZIP 파일을 고른다. 목록의 첫 영상이 자동으로 열린다.
2. 메인 화면의 격자와 실제 인쇄된 글자가 잘 맞는지 확인한다. 스캔이
   기울어져 있다면 **회전 보정 각도**를 조절해 격자가 글자와 일치하도록
   맞춘다 (양수 = 반시계 방향).
   - 격자 시작 위치/칸 크기가 표준과 다르면 **열/행/시작 X/시작 Y/칸
     너비/칸 높이** 값을 직접 조정한다.
3. **폰트 이름**을 입력한다 (예: `THE개이득`). 같은 폰트의 다음 페이지로
   넘어가면 이 값은 자동으로 유지되므로 매번 다시 입력할 필요는 없다.
4. **첫 글자**를 입력한다. 목록의 첫 영상이거나, 바로 앞 영상이 이미
   저장되어 있다면 이 값은 자동으로 채워져 있다 (1.5절 참고). 채워진 값이
   실제 인쇄된 글자와 다르면 직접 수정한다. 입력하면 격자 각 칸 위에
   예상 글자가 파란색으로 표시되므로, 실제 인쇄된 글자와 한 줄씩 눈으로
   대조해 볼 수 있다. 이 값이 맞으면 **마지막 글자**와 총 글자 수가 화면에
   자동으로 계산되어 표시된다 (별도 입력 불필요).
5. **Annotation 저장** 버튼(`Ctrl+S`)을 눌러 저장한다. 저장되면 격자 색이
   초록색으로 바뀌고, 목록의 항목도 초록색으로 바뀐다.
6. **다음 영상으로 이동** 버튼(`Ctrl+N`)을 눌러 다음 영상으로 넘어간다.
   목록에서 항목을 직접 클릭하는 것보다 빠르다.
7. 3~6을 반복한다. 대부분의 영상은 `Ctrl+S` → `Ctrl+N`을 번갈아 누르는
   것만으로 처리된다. `Ctrl+L`을 누르면 저장과 다음 영상 이동을 한 번에
   실행할 수 있다 (저장이 실패하면 다음 영상으로 넘어가지 않는다).

### 1.5 첫 글자 기본값이 자동으로 채워지는 규칙

폰트 하나는 완성형 한글 2,350자를 담기 위해 연속된 5장(500자 x 4장 + 마지막
1장)으로 구성되며, 각 장의 첫 글자는 항상 **가 → 동 → 붰 → 웝 → 탈**
순서로 반복된다 (2.3절 참고).

- 목록의 **첫 번째 영상**은 언제나 새 폰트의 1페이지이므로 첫 글자 기본값은
  항상 `가`이다.
- 그 외의 영상은 **바로 앞 영상의 저장된 annotation**을 읽어, 그 마지막
  글자 다음 글자를 기본값으로 사용한다. (2,350자를 넘어가면 다시 `가`로
  돌아간다.)
- 앞 영상이 아직 저장되지 않았다면 추정할 근거가 없으므로 빈 값으로 둔다.

이 기본값은 어디까지나 **추정값**이며 언제든지 직접 수정(override)할 수
있다. 스캔 누락이나 순서가 어긋난 예외적인 페이지가 있을 수 있으므로,
저장하기 전에 메인 화면의 예상 글자 overlay와 실제 인쇄된 글자를 반드시
눈으로 대조하는 것이 좋다.

### 1.6 키보드 단축키

| 단축키       | 동작                                  |
| ------------ | ------------------------------------- |
| `Ctrl+S`     | Annotation 저장 (저장 버튼과 동일)    |
| `Ctrl+N`     | 다음 영상으로 이동 (이동 버튼과 동일) |
| `Ctrl+L`     | Annotation 저장 후 다음 영상 이동     |
| `Ctrl+←`     | 격자 시작 X를 0.5px 감소              |
| `Ctrl+→`     | 격자 시작 X를 0.5px 증가              |
| `Ctrl+↑`     | 격자 시작 Y를 0.5px 감소              |
| `Ctrl+↓`     | 격자 시작 Y를 0.5px 증가              |
| `Ctrl+Alt+←` | 회전 보정 각도를 0.1도 감소           |
| `Ctrl+Alt+→` | 회전 보정 각도를 0.1도 증가           |

`Ctrl+L`은 저장(`Ctrl+S`)과 다음 영상 이동(`Ctrl+N`)을 순서대로 실행하는
단축키이며, 폰트 이름이나 첫 글자를 입력하지 않아 저장에 실패하면 경고
창만 뜨고 다음 영상으로 넘어가지 않는다.

모든 단축키는 어느 위젯에 포커스가 있어도 동작한다(`bind_all`). 다만 왼쪽
이미지 목록(Listbox)은 `Ctrl+방향키`에 대한 자체 동작을 가지고 있어 이를
가로챌 수 있으므로, 목록에도 같은 단축키를 직접 등록해 목록에 포커스가
있을 때도 정상 동작하도록 했다(2.10절 참고). 격자 위치/회전 보정
단축키는 화살표 키를 여러 번 눌러 격자와 기울기를 실제 인쇄된 글자에
정밀하게 맞출 때 스핀박스를 마우스로 조작하는 것보다 빠르다.

### 1.7 격자 시작 좌표·회전 자동 보정

스캔 과정에서 종이가 밀리면(translate) 칸 크기(너비/높이)는 그대로여도
격자 시작 좌표(시작 X, 시작 Y)만 영상마다 다르게 어긋나는 경우가 있고,
종이가 삐뚤게 놓이면 영상 전체가 조금씩 기울어지는(rotate) 경우도 있다.
`자동 보정` 체크박스(기본 켜짐)가 켜져 있으면, **아직 annotation이
저장되지 않은 영상을 선택할 때마다** 스캔 영상을 분석해 시작 좌표와
회전 보정 각도를 함께 추정한다 (원리는 2.9절 참고).

- 시작 좌표는 표준 기본값 기준 -20~+20px 범위에서 추정한다.
- 회전은 왼쪽/오른쪽, 위/아래 글자들의 치우침 정도가 뚜렷할 때만(대략
  2px 이상 차이) 보정 각도를 계산해 적용한다. 그 정도로 뚜렷하지 않으면
  회전 보정 없이(0.0도) 둔다 — 실제로 기울지 않은 영상에서도 폰트 자체의
  비대칭 잉크 분포 때문에 약간의 차이는 항상 나타나므로, 사소한 차이까지
  회전으로 오인하지 않기 위함이다.
- 이 방식은 대략 2도 이하의 작은 기울기에서 정확하다. 그보다 크게
  기울어진 영상은 이 도구를 사용해 온 경험상 아직 발견되지 않았다.
- 이미 annotation이 저장된 영상에는 적용되지 않는다 (저장된 격자/회전
  값을 그대로 불러온다).
- 자동 추정 값도 결국 추정값이므로, 스핀박스나 `Ctrl+방향키`(격자),
  `Ctrl+Alt+방향키`(회전)로 언제든 미세 조정할 수 있다.
- 체크박스를 끄면 현재 보고 있는(아직 저장되지 않은) 영상의 시작 좌표와
  회전각이 즉시 표준 기본값(0.0도)으로 되돌아가고, 다시 켜면 즉시
  재추정한다.
- `기본값` 버튼은 자동 보정 여부와 무관하게 항상 격자를 표준 기본값으로
  되돌리는 수동 초기화 버튼이다 (회전각은 건드리지 않는다).

## 2부. 상세설계서

### 2.1 원천 데이터 구조 (`data/scan`)

- `data/scan/*.zip`: 각 zip에는 8자리 일련번호(`00000001.jpg` ...)로 된 jpg
  1,000장이 들어 있다 (마지막 zip은 예외적으로 더 적을 수 있다). 일련번호는
  전체 데이터셋에서 겹치지 않는다.
- 영상 한 장 = 폰트 하나의 페이지 한 장. 첫 줄에 폰트 이름이 인쇄되어 있고,
  그 아래 가로 20열 x 세로 25행(500칸) 격자에 글자가 인쇄되어 있다.
- 폰트 하나 = 연속된 5장 (500자 x 4장 + 마지막 1장 ≤ 500자, 합계 2,350자).
  1,000장짜리 zip은 5로 나누어떨어지므로 zip 경계가 항상 폰트 경계와
  일치한다. 즉 어떤 zip을 열어도 **첫 번째 항목은 항상 새 폰트의
  1페이지**이다.
- `data/scan`은 원천 소스이므로 이 도구는 오직 읽기만 하며 절대 수정하지
  않는다.

### 2.2 격자 좌표 모델

```python
DEFAULT_GRID = {
    "cols": 20, "rows": 25,
    "origin_x": 8.0, "origin_y": 241.0,
    "cell_w": 78.76, "cell_h": 78.52,
}
```

이 기본값은 실제 스캔 영상(1654x2338 px)의 픽셀 프로젝션을 분석해 얻은
값이다 (각 행/열의 어두운 픽셀 분포에서 25개 행, 20개 열의 경계를 찾아
평균 칸 간격을 계산). 모든 영상이 같은 인쇄 양식을 쓰므로 대부분의 영상에
그대로 들어맞지만, 스캔 과정에서 생기는 미세한 위치 오차를 보정할 수 있도록
`GridParams`(`cols`, `rows`, `origin_x`, `origin_y`, `cell_w`, `cell_h`)는
화면에서 실시간으로 조정 가능하다.

칸 `(row, col)`의 원본 이미지 좌표 기준 좌상단은 다음과 같이 계산된다.

```text
x = origin_x + col * cell_w
y = origin_y + row * cell_h
```

캔버스에 그릴 때는 여기에 화면 축소/중앙 정렬 배율(`_redraw`가 계산하는
`scale`, `offset_x`, `offset_y`)을 곱해 화면 좌표로 변환한다. 캔버스를
클릭했을 때(`_on_canvas_click`)는 반대로 화면 좌표 → 원본 이미지 좌표 →
`(row, col)`로 역변환하여 상태 표시줄에 보여준다. 이 클릭 검증 기능은 격자
좌표가 실제 인쇄된 글자와 정확히 겹치는지 확인하기 위한 것이다.

### 2.3 완성형 한글 2,350자 순서 (`build_hangul_table`)

annotation은 "첫 글자"만 입력받고 나머지 499자(또는 그 이하)를 표에서
자동으로 채워야 하므로, KS X 1001(완성형) 한글 2,350자의 정확한 순서가
필요하다.

Python 표준 코덱 `euc_kr`/`cp949`는 실제로는 확장된 11,172자(가능한 모든
한글 음절) 전체를 인코딩할 수 있어 그대로 사용할 수 없다. 대신
`iso2022_kr` 코덱은 KS X 1001이 정의한 2,350자만 인코딩을 허용하므로, 이를
필터로 사용해 순서를 만든다.

```python
table = []
for code in range(0xAC00, 0xD7A4):       # 유니코드 한글 음절 전체
    ch = chr(code)
    try:
        ch.encode("iso2022_kr")           # 완성형 2,350자만 성공
    except UnicodeEncodeError:
        continue
    table.append(ch)
```

이 방식으로 만든 순서가 실제 인쇄 순서와 정확히 일치하는지 `sample.jpg`와
`001.zip`의 첫 페이지를 잘라 육안으로 대조하여 확인했다. 특히 각 페이지가
500자 단위로 나뉘므로, `table[0]`, `table[500]`, `table[1000]`,
`table[1500]`, `table[2000]`이 각각 **가, 동, 붰, 웝, 탈**임을 확인했으며,
이는 폰트 한 개를 구성하는 5개 페이지의 시작 글자와 정확히 일치한다
(1.5절의 자동 기본값 추정 로직이 이 사실에 기반한다).

### 2.4 회전 보정

스캔 영상이 기울어진 경우를 위해 `rotation_deg`(도 단위, 양수 = 반시계
방향)를 입력받는다. 화면에는 `Image.rotate(angle, resample=BICUBIC,
expand=False, fillcolor=흰색)`으로 회전한 이미지를 표시한다.
`expand=False`를 사용하므로 회전해도 이미지 크기(가로/세로 픽셀 수)는
바뀌지 않고, 격자 좌표(`origin_x/y`, `cell_w/h`)는 "회전 보정이 끝난 뒤의
이미지" 좌표계를 기준으로 그대로 적용된다. 즉 사용자는 먼저 회전각으로
기울기를 바로잡은 뒤, 격자를 표준 위치에 맞추는 순서로 작업한다.

회전된 이미지는 `_rotated_cache`에 `(각도, 원본 이미지, 회전된 이미지)`로
캐시되어, 같은 각도로 반복 redraw(창 크기 조절 등)할 때 매번 다시 계산하지
않는다.

### 2.5 annotation 자동 기본값 (`_guess_first_char`)

`_apply_annotation_fields`는 이미지를 선택할 때마다 호출되며, 그 영상에
저장된 annotation이 있으면 그대로 불러오고, 없으면 다음 규칙으로 입력란을
초기화한다.

- **폰트 이름**: 초기화하지 않는다 (그대로 유지 → 연속된 페이지에서 반복
  입력 방지).
- **회전 보정 각도, 격자 좌표**: 매 영상마다 다를 수 있으므로 표준
  기본값(`DEFAULT_GRID`, `0.0`)으로 초기화한다.
- **첫 글자**: `_guess_first_char(index)`가 계산한다.
  - `index == 0` (목록의 첫 영상) → `hangul_table[0]` (`가`).
  - 그 외 → `data/annotation/<바로 앞 영상 8자리번호>.json`을 읽어
    `last_char_index + 1`을 다음 글자 인덱스로 사용한다. 2,350을 넘으면
    `0`으로 순환한다 (새 폰트 시작).
  - 앞 영상이 저장되어 있지 않으면 빈 문자열을 반환해 사용자가 직접
    입력하도록 한다.

이 로직은 "페이지 번호 % 5"처럼 목록 위치에만 의존하는 방식이 아니라
**직전 영상에 실제로 저장된 값**에 의존한다. 따라서 중간에 스캔이 누락되거나
순서가 어긋난 예외적인 페이지가 있어도, 그 지점 이후부터는 사용자가 값을
직접 override하면 이후 페이지의 추정이 다시 정상적으로 이어진다.

### 2.6 annotation 파일 형식

- 저장 위치: `data/annotation/<jpg 8자리 파일명>.json` (예:
  `data/annotation/00000001.json`). zip 안의 실제 경로(`entry`)가 아니라
  jpg 파일명만으로 키를 만들며, 파일명은 전체 데이터셋에서 겹치지 않는다.
- 인코딩: UTF-8, `ensure_ascii=False`로 한글을 그대로 저장한다.

실제 저장 예시:

```json
{
  "zip": "001.zip",
  "entry": "001/00000001.jpg",
  "image_name": "00000001.jpg",
  "image_width": 1654,
  "image_height": 2338,
  "font_name": "THE개이득",
  "first_char": "가",
  "first_char_index": 0,
  "last_char": "돗",
  "last_char_index": 499,
  "char_count": 500,
  "rotation_deg": 0.0,
  "grid": {
    "cols": 20,
    "rows": 25,
    "origin_x": 8.0,
    "origin_y": 241.0,
    "cell_w": 78.76,
    "cell_h": 78.52
  }
}
```

| 필드                              | 설명                                                                                                     |
| --------------------------------- | -------------------------------------------------------------------------------------------------------- |
| `zip`, `entry`, `image_name`      | 원본 위치 추적용 (어떤 zip의 어떤 경로인지)                                                              |
| `image_width`, `image_height`     | 원본 영상 픽셀 크기                                                                                      |
| `font_name`                       | 사용자가 입력한 폰트 이름                                                                                |
| `first_char` / `first_char_index` | 좌상단 칸의 글자와 2,350자 표 안에서의 인덱스(0-base)                                                    |
| `last_char` / `last_char_index`   | 격자 크기로부터 자동 계산된 마지막 칸의 글자와 인덱스                                                    |
| `char_count`                      | 이 영상에 실제로 들어있는 글자 수 (`min(cols*rows, 남은 글자 수)`, 마지막 페이지는 500보다 작을 수 있음) |
| `rotation_deg`                    | 회전 보정 각도 (도, 양수 = 반시계)                                                                       |
| `grid`                            | 이 영상에 적용된 격자 좌표 (`cols`, `rows`, `origin_x`, `origin_y`, `cell_w`, `cell_h`)                  |

이 정보만으로 향후 별도 스크립트가 (1) 회전 보정 → (2) 격자대로 500칸
잘라내기 → (3) `first_char_index`부터 순서대로 라벨링, 까지 재현할 수 있다.

### 2.7 저장 검증 규칙 (`_on_save_clicked`)

다음 조건을 만족해야 저장되며, 하나라도 어긋나면 경고창을 띄우고 저장하지
않는다.

1. 이미지가 선택되어 있어야 한다.
2. 폰트 이름이 비어 있으면 안 된다.
3. 첫 글자가 완성형 2,350자 표에 있는 글자여야 한다 (`char_index`에 존재).

### 2.9 격자 시작 좌표·회전 자동 보정 알고리즘 (`_estimate_origin_and_rotation`)

annotation이 없는 영상을 선택하면 `_apply_annotation_fields`가 격자를
`DEFAULT_GRID`로, 회전각을 0.0으로 초기화한 뒤, `자동 보정`
체크박스(`auto_correct_var`)가 켜져 있으면 `_estimate_origin_and_rotation`
으로 시작 좌표(`origin_x`, `origin_y`)와 회전 보정 각도(`rotation_deg`)를
함께 추정해 덮어쓴다. 칸 크기(`cell_w`, `cell_h`)는 정확하다고 가정하고
건드리지 않는다. 시작 좌표는 1차(위상 분석)와 2차(바운딩 박스) 두 단계로,
회전은 2차 보정에서 이미 계산한 칸별 데이터를 재사용해 세 번째 단계로
추정한다(`_estimate_origin_stage`가 1차+2차를, `_estimate_rotation_deg`가
회전을 담당하고, `_estimate_origin_and_rotation`이 이들을 오케스트레이션
한다).

#### 2.9.1 1차 보정: 잉크 프로젝션 위상 분석

원리는 다음과 같다.

1. 영상을 흑백으로 변환하고 밝기 200 미만인 픽셀을 "잉크"로 간주해
   이진화한다.
2. 세로 방향으로 픽셀을 합산해 **열(column) 프로젝션**을, 가로 방향으로
   합산해 **행(row) 프로젝션**을 만든다. 이때 각 프로젝션은 반대쪽 축의
   전체 범위가 아니라 `DEFAULT_GRID` 기준 격자 영역(±25px 여유)만
   사용해, 격자 밖 영역이 프로젝션을 오염시키지 않게 한다. 다만 행
   프로젝션(origin_y 추정용)은 열 방향으로 격자의 가로 폭 전체를
   사용하므로 영상 맨 위 폰트 이름 인쇄 영역(가로로 격자 폭 안쪽에
   있음)까지 함께 더해진다. 이 영역은 격자와 무관하므로
   `TITLE_AREA_HEIGHT`(180px) 안쪽 프로젝션 값을 0으로 지운 뒤 위상을
   계산한다.
3. 글자가 20열(또는 25행) 간격으로 반복 인쇄되어 있으므로, 프로젝션은
   `cell_w`(또는 `cell_h`)를 주기로 하는 준-주기 신호가 된다. 이 신호에서
   주기 `cell_w`에 해당하는 이산 푸리에 계수
   `Σ profile(x)·e^(-i·2π·x/cell_w)`를 계산하면, 그 위상(phase)이 바로
   프로젝션이 가장 밝은(잉크가 몰린) 위치, 즉 칸 **중심(peak)**의 정확한
   좌표를 알려준다. 얇은 구간을 표본 추출하는 방식보다 프로젝션 전체를
   사용하므로 개별 글자 획의 잡음에 훨씬 덜 민감하다.
4. 칸 중심에서 반 칸을 빼면 칸 **경계(valley)**, 즉 격자 시작 좌표 후보를
   얻는다. 이 후보는 주기 `cell_w`마다 반복되므로, `DEFAULT_GRID`의
   시작 좌표에 가장 가까운 후보를 고르고, 사용자가 지정한 탐색 범위
   (`AUTO_CORRECT_RANGE`, 기본 -20~+20px)를 벗어나면 그 경계값으로
   자른다.
5. X, Y축을 각각 독립적으로 위 과정을 수행한다 (X 추정에는 Y의
   `DEFAULT_GRID` 값을, Y 추정에는 X의 `DEFAULT_GRID` 값을 프로젝션
   범위를 자르는 데만 사용하므로 서로 크게 간섭하지 않는다).

001.zip의 첫 페이지처럼 실제 좌표를 별도로 측정해 둔 영상으로 검증한
결과, 이 방법은 1px 미만의 오차로 시작 좌표를 추정했다 (초기 구현에서
시도했던, 경계/중심 근방 몇 픽셀만 표본 추출해 평균을 비교하는 방식은
같은 영상에서 6px 이상 어긋나는 경우가 있어 폐기했다).

폰트 이름 인쇄 영역을 지우는 보정(`TITLE_AREA_HEIGHT`)의 실제 효과는
여러 zip(001/005/010/018)의 샘플로 측정했을 때 origin_y 추정값을
0.2~0.3px 정도만 바꾸는 수준이었다 — 폰트 이름의 잉크량이 훨씬 많은
폰트(005.zip)에서도 비슷했다. 즉 이 보정은 원칙적으로는 맞지만 큰 폭의
어긋남(2.9.2절)을 설명하는 요인은 아니었다.

#### 2.9.2 왜 1차 보정(위상 분석)만으로는 부족한가

`002.zip`의 `00001513.jpg`는 1차 보정만으로는 origin_y가 탐색 범위 경계
(`DEFAULT_GRID` 대비 -20px, 즉 221.0)에 그대로 붙어버렸고, 격자를 그려보면
글자가 각 칸의 위쪽에 몰리고 아래쪽에 큰 여백이 남는 것이 뚜렷하게
보였다. "그냥 해보니 바운딩 박스 방식이 더 잘 맞더라"에 그치지 않고, 왜
위상 분석만으로 부족한지 두 가지 원인으로 분해해서 확인했다.

**원인 1 (주된 원인): 탐색 범위(±20px)가 실제 폰트별 인쇄 위치 편차보다
좁다.** `AUTO_CORRECT_RANGE`의 클램프를 걷어내고 순수 위상 계산 결과만
보면, `00001513.jpg`는 원래 -28.2px 이동을 요구했다(`origin_y=212.78`).
탐색 범위가 ±20px로 제한되어 있어 -28.2px 요구가 경계값 -20px로 잘려
221.0으로 표시된 것이었지, 위상 분석 자체가 엉뚱한 값을 계산한 것이
아니었다. 같은 방식으로 다른 이미지도 확인해 보면 `018.zip`은
-25.1px, `010.zip`은 -19.8px를 요구해 다들 경계 근처이거나 넘어선다.
±20px는 스캔 과정에서 종이가 밀리는 정도의 오차를 염두에 두고 정한
값인데, 폰트 자체가 칸 안에서 위/아래로 치우쳐 인쇄되는 정도(활자
디자인/메트릭 문제)는 이 가정보다 클 수 있다는 뜻이다.

**원인 2 (부차적 원인, 보통 1~2px 수준): 위상 분석이 찾는 위치는
"바운딩 박스 중심"이 아니라 "잉크 밀도로 가중한 무게중심"이다.** 클램프를
없앤 순수 위상 값도 바운딩 박스 결과와 정확히 일치하지는 않는다
(`00001513.jpg`는 -1.6px, `010.zip`은 -1.8px, `018.zip`은 -1.2px, 반면
`001/005.zip`은 0.1~0.3px로 거의 일치). 실제로 개별 칸에서 두 값을
직접 비교해 보면(`010.zip` 여러 칸), 바운딩 박스 중심은 칸 중심보다
평균 2px 정도 위에 있는데 잉크 무게중심은 그보다 더 위(칸 중심 대비
2.6~5.2px)에 있었다. 이는 한글 음절 특성상 초성+중성이 모인 위쪽
획이 받침이 있는 아래쪽보다 잉크가 진하고 많은 경우가 흔해서, 잉크
양으로 가중한 무게중심이 글자의 실제 시각적 범위(바운딩 박스)보다 위로
쏠리기 때문이다. 위상 분석은 근본적으로 (하나의 주파수 성분에 대한)
가중 무게중심을 구하는 방법이므로, 이 편차는 알고리즘을 아무리
정교화해도 없앨 수 없고 애초에 다른 통계량이다.

두 원인 모두 근거는 있지만 성격이 다르다. 원인 1은 파라미터(탐색 범위)의
한계이고, 원인 2는 통계량 자체의 차이다. 바운딩 박스 기반 2차 보정은 이
둘을 동시에 해결한다 — 탐색 범위에 묶이지 않고, 애초에 사람이 "격자가
글자에 맞다"고 판단할 때 기준으로 삼는 바로 그 통계량(바운딩 박스
중심)을 직접 측정하기 때문이다. 1차 보정의 ±20px 클램프 자체는 그대로
유지한다 — 스캔 잡음에 대한 보수적인 안전장치 역할은 여전히 유효하고,
더 큰 보정이 필요한 경우는 2차 보정이 실측으로 담당하므로 1차 보정의
탐색 범위를 넓힐 필요가 없다.

#### 2.9.3 2차 보정: 칸별 글자 바운딩 박스 (`_refine_origin_by_bbox`)

2차 보정은 다음과 같이 동작한다.

1. 1차 보정으로 구한 `origin_x`, `origin_y`로 20x25 격자를 나눈다.
2. 각 칸마다 그 안의 잉크 픽셀만 모아 바운딩 박스(최소/최대 x, y)를
   구하고, 그 중심과 칸의 기하학적 중심의 차이(`dx`, `dy`)를 계산한다.
   잉크 픽셀이 4개 이하인 칸(마지막 페이지의 빈 칸 등)은 건너뛴다.
3. 유효한 칸이 전체의 30% 미만이면 (거의 빈 페이지 등 신뢰할 수 없는
   경우) 2차 보정을 생략하고 1차 결과를 그대로 반환한다.
4. 그렇지 않으면 모든 칸의 `dx`, `dy` 평균을 구하고, 그 값만큼
   `origin_x`, `origin_y`를 한 번 더 옮긴다. 극단값 방지를 위해 평균
   보정량은 칸 크기의 절반(`±cell_w/2`, `±cell_h/2`)으로 제한한다.

2차 보정은 1차 보정처럼 `AUTO_CORRECT_RANGE`(-20~+20px)에 묶이지 않는다.
칸 안의 실제 잉크 위치를 직접 측정한 값이므로, 필요하다면 그보다 더 크게
옮길 수 있어야 하기 때문이다. 실제로 `00001513.jpg`는 1차 보정만으로는
origin_y가 221.0(경계에 클램프됨)에 머물렀지만, 2차 보정을 거치면
214.4까지 더 이동하며 격자가 글자 중심에 정확히 들어맞았다. 이 방법으로
`005/010/018.zip`의 서로 다른 폰트에서도 모두 격자가 글자 중심에 맞는
것을 육안으로 확인했다. 계산 비용은 칸 500개를 순회해도 이미지 한 장당
약 35ms 수준이라 자동 보정 체감 속도에 영향을 주지 않는다.

#### 2.9.4 회전 보정 각도 추정 (`_estimate_rotation_deg`)

**원리.** 영상이 작은 각도 θ만큼 기울어져 있으면(라디안, 작은 각도
근사), 회전 중심 (cx, cy)에서 (x₀, y₀)만큼 떨어진 글자의 위치는 대략
`Δx ≈ -θ·(y₀-cy)`, `Δy ≈ +θ·(x₀-cx)`만큼 어긋난다. 즉 **세로 편차는
가로 위치에 비례**하고 **가로 편차는 세로 위치에 비례**한다. 그래서
2.9.3절의 칸별 바운딩 박스 편차(`_collect_cell_offsets`가 이미 계산해
둔 `dx`, `dy`)를 칸의 왼쪽 절반/오른쪽 절반, 위쪽 절반/아래쪽 절반으로
나눠 평균을 비교하면 θ를 두 축에서 독립적으로 추정할 수 있다.

```text
θ_from_y ≈ (dy_오른쪽평균 - dy_왼쪽평균) / (x_오른쪽평균 - x_왼쪽평균)
θ_from_x ≈ -(dx_아래평균 - dx_위평균) / (y_아래평균 - y_위평균)
```

두 그룹으로 나눠 평균을 내는 것은 왼쪽/오른쪽(또는 위/아래) 모두에
똑같이 실려 있는 "폰트 고유의 칸 내 치우침"(2.9.2절의 원인 2)을 차이로
상쇄시키는 효과도 있어, 회전이 아닌 요인이 섞여 들어가지 않는다.

**검증.** 실제 스캔 영상을 알려진 각도(0.5°, 1.0°, 2.0°, -1.5°)로
합성 회전시킨 뒤 이 공식으로 역으로 추정해 보면, ±1.5° 범위에서는 원래
각도를 0.1° 안팎의 오차로 복원했다(부호도 그대로 `rotation_deg`에 넣으면
바로 교정되도록 맞춰져 있다 — 추가 부호 반전이 필요 없음을 실제 이미지
회전으로 확인했다). 다만 2.0°에서는 정확도가 눈에 띄게 떨어졌는데, 이
각도에서는 영상 가로 폭(약 1575px) 기준 좌우 끝의 세로 이동량이 반 칸
(약 39px)을 넘어서면서 글자가 이웃 칸으로 넘어가 칸별 바운딩 박스
측정 자체가 부정확해지기 때문이다. 실제 스캔에서는 아직 2° 이상 기울어진
영상이 발견되지 않았으므로, 이 오차는 현재로서는 실용적인 한계는 아니다.

**동작 방식.**

1. 칸이 30% 미만으로 유효하면(거의 빈 페이지 등) `None`을 반환해
   회전 보정을 하지 않는다(2차 보정과 같은 최소 표본 기준을 공유한다).
2. 왼쪽/오른쪽, 위/아래 각 그룹이 5칸 미만이면 역시 `None`을 반환한다.
3. 왼쪽-오른쪽 dy 평균 차이와 위-아래 dx 평균 차이가 **둘 다**
   `ROTATION_BIAS_THRESHOLD_PX`(기본 2px) 미만이면 회전이 없다고 보고
   `None`을 반환한다. 기울지 않은 영상에서도 폰트 고유의 비대칭 잉크
   분포 때문에 1~3px 정도 차이는 흔히 나타나므로, 이를 회전으로 오인하지
   않기 위한 문턱값이다.
4. 문턱값을 넘으면 위 공식으로 θ_from_y, θ_from_x를 계산해 평균한
   각도(도 단위)를 반환한다.

**두 단계 구조와 비용.** `_estimate_origin_and_rotation`은 먼저 회전
없이(0도) `_estimate_origin_stage`를 실행해 시작 좌표와 칸별 편차를
구하고, 그 편차로 `_estimate_rotation_deg`를 호출한다 — 이미 계산해 둔
값을 재사용하므로 픽셀을 다시 훑지 않아 사실상 비용이 들지 않는다. 회전이
없다고 판단되면(`None`) 그 결과를 그대로 반환한다. 회전이 있다고
판단되면 그제서야 그 각도로 영상을 회전시키고 `_estimate_origin_stage`를
**한 번 더** 실행해, 회전이 반영된 상태에서 시작 좌표를 다시 구한다. 실제
비용이 드는 부분은 이 재실행이 아니라 `Image.rotate()` 자체다(1654x2338
BICUBIC 회전에 약 130ms) — 그래서 회전이 필요 없는(대다수) 영상은
1차+2차 보정만 수행하는 기존과 같은 속도(약 35ms)를 유지하고, 회전이
필요한 영상만 추가 비용(약 200ms)을 치른다. 이때 계산한 회전된 이미지는
`self._rotated_cache`에 바로 저장해 두므로, 화면을 다시 그릴 때
`_get_display_image`가 같은 각도로 또 회전시키지 않는다.

### 2.10 전역 단축키가 Listbox에서도 동작하도록 만든 이유

Tk의 `Listbox`는 방향키(그리고 일부 `Ctrl+방향키` 조합)에 대한 자체 클래스
바인딩을 가지고 있다. Tkinter의 이벤트 처리 순서는
`(위젯 자신 → 위젯 클래스 → 최상위 창 → bind_all)` 순이며, 앞 단계의
바인딩이 `"break"`를 반환하면 뒤 단계로 이벤트가 전달되지 않는다. 그
결과 왼쪽 이미지 목록에 포커스가 있는 상태에서 `Ctrl+↑`/`Ctrl+↓`를
누르면 Listbox의 기본 동작이 이벤트를 먼저 가로채 `_bind_shortcuts`가
`self`(최상위 창)에 등록한 `bind_all` 핸들러까지 도달하지 못하는
문제가 있었다.

해결 방법은 같은 단축키를 `self.image_listbox`에도 직접 등록하고, 그
핸들러가 항상 `"break"`를 반환하도록 만드는 것이다. 위젯 자신에게 등록한
바인딩은 클래스 바인딩보다 먼저 실행되므로, 목록에 포커스가 있어도 우리
핸들러가 먼저 실행되고 `"break"`로 Listbox의 기본 동작을 차단한다.
`_bind_shortcuts`는 이 등록을 6개의 격자/회전 단축키 모두에 동일하게
적용한다.

### 2.11 알려진 제한사항 및 향후 과제

- 격자/회전 값은 영상 단위로만 저장되며, 폰트 전체(5장)에 일괄 적용하는
  기능은 없다. 페이지마다 스캔 상태가 달라 의도적으로 그렇게 설계했다
  (README의 진행 상황 참고).
- zip 파일을 바꿀 때 이전 zip의 `zipfile.ZipFile` 핸들은 닫히지만, 실행 중
  하나의 zip 핸들만 유지한다 (여러 zip을 동시에 열어두지 않음).
- `data/annotation`에 저장된 값을 이용해 실제로 글자 이미지를 잘라내고
  학습 데이터셋을 만드는 단계는 아직 이 도구의 범위가 아니다 (README의
  "진행 상황" 참고).
- 격자 시작 좌표·회전 자동 보정(2.9절)은 칸 크기(`cell_w`/`cell_h`)가
  맞다는 가정 위에서 이동(translate)과 작은 회전(rotate) 오차만
  보정한다. 회전 추정은 대략 2도 이하에서 정확하며, 그보다 크게
  기울어진 영상(2.9.4절 참고)이나 -20~+20px 범위 밖으로 크게 어긋난
  영상, 잉크가 매우 적은/많은 특수한 폰트에서는 추정이 부정확할 수
  있으므로 항상 격자 overlay로 눈으로 확인해야 한다.
