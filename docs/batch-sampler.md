# Font Group Batch Sampler — 사용 설명서 및 상세설계서

`font_classifier/batch_sampler.py`의 `FontGroupBatchSampler`는
`FontGlyphDataset`(dataset_loader.py)과 짝을 이루는 `DataLoader` 배치
샘플러다. GUI 도구가 아니라 학습 스크립트가 `import`해서 쓰는 라이브러리
모듈이므로 `bin\*.bat` 런처는 없다.

이 문서는 두 부분으로 구성된다.

- **1부. 사용법** — 학습 스크립트에서 이 샘플러를 쓰는 방법
- **2부. 상세설계서** — 왜 필요한지, K/M 기본값 근거, 알고리즘, 트레이드오프

---

## 1부. 사용법

### 1.1 기본 사용

```python
from torch.utils.data import DataLoader
from font_classifier.dataset_loader import FontGlyphDataset
from font_classifier.batch_sampler import FontGroupBatchSampler

train_ds = FontGlyphDataset()
sampler = FontGroupBatchSampler(train_ds)          # 기본값 K=32, M=8 (배치 크기 256)
loader = DataLoader(train_ds, batch_sampler=sampler, num_workers=4)

for epoch in range(num_epochs):
    for batch in loader:               # 매 epoch(= for 루프 진입)마다 폰트 묶음이 새로 섞인다
        ...
```

`batch_sampler=`를 넘길 때는 `DataLoader`의 `batch_size`/`shuffle`/`sampler`
인자를 **함께 쓰면 안 된다** — 배치 크기와 순서 전체를
`FontGroupBatchSampler`가 결정한다. `num_workers>0`을 Windows에서 쓰려면
학습 스크립트를 `if __name__ == "__main__":` 블록 안에서 실행해야 한다
([dataset-loader.md](dataset-loader.md) 1.2절과 동일한 PyTorch 요구사항).

### 1.2 K/M 조정

```python
sampler = FontGroupBatchSampler(train_ds, fonts_per_batch=16, chars_per_font=4)
```

- `fonts_per_batch`(K): 배치 하나에 섞이는 서로 다른 폰트(=분류 클래스)
  수. 크게 잡을수록 폰트 분류 손실의 배치당 클래스 다양성이 좋아지지만,
  캐시에 동시에 붙잡아 둬야 하는 폰트 수도 늘어난다([dataset-loader.md](dataset-loader.md)
  1.5절의 `max_cache_bytes`와 함께 조정한다 — 최소 `K × 9.18MiB` 이상은
  되어야 그 폰트 묶음이 캐시에서 밀려나지 않는다).
- `chars_per_font`(M): 폰트 하나당 배치에 들어가는 글자 수. 배치 크기는
  `K * M`이다(마지막 폰트 묶음/폰트별 마지막 라운드는 더 작을 수 있다 —
  1.3절).
- 기본값(K=32, M=8)의 근거는 2.2절 참고.

### 1.3 배치 크기가 항상 K*M은 아니다

폰트 수(K로 나누어떨어지지 않음)나 폰트별 유효 글자 수(M으로
나누어떨어지지 않음, 그리고 폰트마다 빈 칸 개수가 조금씩 다름) 때문에,
마지막 폰트 묶음이나 폰트별 마지막 라운드는 `K*M`보다 작은 배치를 만들
수 있다. 기본값 `drop_last=False`는 이런 배치도 그대로 내보내
(가변적인 배치 크기를 감수하고) 데이터를 하나도 버리지 않는다.
`drop_last=True`로 만들면 정확히 `K*M` 크기인 배치만 내보낸다(그만큼
매 epoch 일부 글자가 누락된다).

### 1.4 `len(sampler)`는 근사치다

`FontGroupBatchSampler.__len__`(진행률 표시줄 등에 쓰인다)은 폰트별 유효
글자 수가 조금씩 다르고 폰트 묶음 구성이 매번 무작위로 바뀌기 때문에
**정확한 배치 수가 아니라 근사치**다. 계산 방법은 2.4절 참고.

## 2부. 상세설계서

### 2.1 왜 필요한가 — plain `shuffle=True`는 캐시를 무력화한다

`FontGlyphDataset`의 LRU 캐시([dataset-loader.md](dataset-loader.md) 2.3절)는
같은 폰트가 짧은 시간 안에 반복 조회되어야 값어치가 있다. 그런데 폰트가
약 3,480개나 되는 이 데이터셋에서 완전히 무작위인 `shuffle=True`를 쓰면:

- 배치 하나(예: 64개 표본)를 만들 때, 생일 문제(birthday paradox) 근사로
  거의 항상 64개 **전부 다른 폰트**가 뽑힌다(폰트 수가 배치 크기보다
  훨씬 많으므로 충돌이 드물다).
- 폰트 하나가 평균 ~2,300자를 갖고 전체 표본이 ~700만 개라면, 같은
  폰트를 다시 방문할 때까지의 평균 간격은 약 `7,000,000 / 2,300 ≈
  3,043`개 표본이다. 그 사이에 수천 개의 **다른** 폰트가 캐시를 스쳐
  지나가므로, 캐시 용량이 전체 폰트 수(~3,480개, 약 32GiB)에 근접하지
  않는 한 재방문 전에 반드시 밀려난다.

즉 plain `shuffle=True`에서는 LRU 캐시가 사실상 매번 캐시 미스만
겪는다 — 글자 하나(4KB)를 위해 9MB짜리 PNG 전체를 다시 열고 디코딩하는
셈이라, 캐시가 없는 것과 거의 같아진다.

### 2.2 해법 — "PK 샘플링"과 K/M 기본값 근거

해법은 한 번에 소수의 폰트(K개)만 활성화해 여러 배치에 걸쳐 재사용하는
것이다. 이는 metric learning(얼굴 인식, person re-id 등)에서 흔히 쓰는
**"PK 샘플링"**(P개 클래스 x 클래스당 K개 샘플로 배치를 구성하는 방식,
이 문서의 표기로는 폰트 수 K와 폰트당 글자 수 M에 대응)과 완전히 같은
발상이다 — 대규모 클래스 분류/embedding 학습에서 이미 널리 검증된
패턴이므로, 이 프로젝트에서 처음 시도하는 위험한 기법이 아니다.

중요한 점: **M은 캐시 적중률에 영향을 주지 않는다.** 폰트가 캐시에
남아 있는 한, 디코딩 1회는 그 폰트의 유효 글자 수 전체(M과 무관하게
~2,300자)에 상각된다. M을 8에서 4로 줄이면 그 폰트가 만들어내는 배치
수만 늘어날 뿐, 캐시 적중률 자체는 그대로다. 그래서 두 파라미터는
서로 다른 것을 책임진다.

- **K(`fonts_per_batch`)가 캐시 적중률과 폰트 분류 손실의 배치당 클래스
  다양성을 함께 책임진다.** K가 작을수록(극단적으로 K=1) 캐시 적중률은
  최고가 되지만, 배치 하나가 사실상 폰트 1종의 softmax 손실만 반영해
  그래디언트가 그 한 클래스에 쏠린다 — 대규모 다중 클래스 분류에서
  잘 알려진 저다양성 배치 문제다. K가 클수록 클래스 다양성은 좋아지지만
  캐시에 동시에 붙잡아 둬야 하는 폰트 수도 늘어난다(다만 폰트 하나가
  ~9.18MiB에 불과해 수백 개를 캐시해도 수백 MiB~수 GiB 수준이라, 실제
  제약은 K가 아니라 `max_cache_bytes` 설정이다).
- **M(`chars_per_font`)이 배치 크기와, 같은 폰트끼리의 쌍(pair) 가용성을
  책임진다.** [model-design.md](model-design.md) 4.2~4.4절의 대조학습/
  재구성 손실은 배치 안에서 같은 폰트의 서로 다른 글자 쌍(style-positive)
  을 필요로 한다 — M이 클수록 폰트 하나당 그런 쌍이 `C(M,2)`개씩
  늘어난다.

폰트 약 3,480개, 폰트당 최대 2,350자(construct-dataset.py의 실패 임계값
때문에 실제로는 대부분 2,346자 이상)라는 실제 규모에서, 기본값을
다음과 같이 정했다.

| 파라미터             | 기본값 | 근거                                                                                                                                                                                                       |
| -------------------- | ------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `fonts_per_batch`(K) | 32     | 배치마다 서로 다른 폰트 클래스 32개(전체 3,480종의 약 0.9%)가 섞여, 대규모 PK 샘플링 사례(보통 P=16~128 범위)에 부합하는 수준의 클래스 다양성을 준다. 캐시 비용은 `32 x 9.18MiB ≈ 294MiB`로 무시할 만하다. |
| `chars_per_font`(M)  | 8      | 배치 크기 `K*M=256`(64x64 그레이스케일에 넉넉한 크기)을 만들고, 폰트 하나당 `C(8,2)=28`쌍의 동일 폰트 쌍을 확보해 Phase 3 대조학습/재구성 손실에도 바로 쓸 수 있게 여유를 둔다.                            |

두 값 모두 생성자 인자로 바꿀 수 있으며(1.2절), 절대적인 정답이라기보다
이 데이터셋 규모에 맞춘 합리적인 출발점이다.

### 2.3 알고리즘 (`__iter__`)

매 epoch(= `__iter__` 호출)마다:

1. 유효 글자가 하나 이상 있는 폰트 목록(`_font_positions`)을 무작위로
   섞는다.
2. `fonts_per_batch`(K)개씩 연속으로 묶어 폰트 묶음을 만든다(마지막
   묶음은 `drop_last=False`면 K보다 작을 수 있다).
3. 묶음 안의 폰트마다: 그 폰트의 유효 글자 인덱스 목록을 무작위로 섞고
   `chars_per_font`(M)개씩 잘라 "라운드" 목록을 만든다(마지막 라운드는
   M보다 작을 수 있다).
4. 라운드 번호 0, 1, 2, ...  순서로, 그 라운드가 아직 남아 있는 모든
   폰트의 몫을 모아 배치 하나를 만들어 낸다. 즉 같은 K개 폰트가
   `max(그 묶음 폰트들의 라운드 수)`개의 연속된 배치에 걸쳐 재사용된다.
5. 다음 폰트 묶음으로 넘어가 2~4를 반복한다.

`drop_last=True`면 4단계에서 정확히 `K*M` 크기가 아닌 배치를 건너뛴다.

이 순서 덕분에, 한 폰트 묶음이 활성화되어 있는 동안(=여러 연속된
배치 동안) 그 K개 폰트는 캐시에 계속 남아 있고, 각 배치는 매번 그
묶음 전체(또는 이미 소진되지 않은 나머지)를 활용한다 — 캐시 적중률과
배치 다양성을 동시에 만족시키는 지점이다.

### 2.4 `__len__` 근사 계산

폰트 묶음 하나가 만들어내는 배치(라운드) 수는 **그 묶음의 폰트
개수가 아니라 폰트당 유효 글자 수**로 정해진다 — K보다 작은 마지막
묶음도 다른 묶음과 똑같은 라운드 수를 만들어 내고(다만 라운드마다
배치가 더 작을 뿐이다), 그래서 단순히 "전체 유효 글자 수 ÷ (K*M)"으로
나누면 폰트 수가 K로 나누어떨어지지 않을 때 크게 틀린다(개발 중 실측:
폰트 40개·K=32 설정에서 이 단순 계산은 358이라는 값을 냈지만 실제
배치 수는 572였다 — 약 40% 과소평가).

대신 "묶음 수 x 묶음당 평균 라운드 수"로 근사한다.

```python
num_groups = ceil(유효 폰트 수 / K)
avg_chars = 유효 폰트들의 평균 유효 글자 수
rounds_per_group = ceil(avg_chars / M)
근사 배치 수 = num_groups * rounds_per_group
```

폰트별 유효 글자 수 편차(대부분 2,346~2,350자, [construct-dataset.md](construct-dataset.md)
2.3절의 실패 임계값 때문에 편차가 최대 4~5자 수준)가 작아 이 근사는
실제 값과 거의 일치한다(합성 테스트 데이터로 정확히 일치함을 확인
했다). 폰트 40개·K=32·M=8 조건에서 실제 배치 수(572)와 정확히 일치함을
직접 확인했다.

### 2.5 알려진 제한사항

- **정확히 균등하지 않은 배치 크기**: 1.3절 — `drop_last=False`(기본값)
  에서는 배치 크기가 `M`(폰트 묶음의 마지막 라운드, 폰트 1개분)부터
  `K*M`까지 다양할 수 있다. 고정 크기 배치가 필요하면 `drop_last=True`를
  쓴다(그만큼 매 epoch 일부 글자가 누락된다 — 폰트당 최대 `M-1`자, 무시할
  수준).
- **`model-design.md` 4.3절의 "그리드"(같은 M개 글자를 K개 폰트에 공통으로
  적용)는 구현하지 않았다**: 이 샘플러는 폰트마다 **독립적으로** 무작위
  M개 글자를 뽑는다. 4.3절이 말하는, 같은 글자가 여러 폰트에 공통으로
  나타나는 content-positive 열(column) 쌍은 만들어지지 않는다 —
  지금 필요한 것(캐시 지역성 + 분류 손실의 배치당 클래스 다양성)에는
  이 정도로 충분하고, Phase 3에서 실제로 열 공유 그리드가 필요해지는
  시점에 확장하는 편이 낫다고 판단했다(그때 가서야 구멍(빈 칸) 처리
  방식 등 그리드 특유의 설계 질문에 답할 수 있다).
- **재현 가능한 seed 제어 없음**: `__iter__`가 Python 표준 `random`
  모듈의 전역 상태를 그대로 쓴다 — [dataset-loader.md](dataset-loader.md)
  2.8절과 같은 이유로 지금은 다루지 않는다.
- **`len()`은 근사치다**(2.4절) — 폰트 묶음 구성이 무작위이므로 실제
  배치 수가 근사치와 정확히 일치하지 않을 수 있다(다만 실측상 편차는
  아주 작다).
